# Минималистичный сайт на Node.js + Express + SQLite (light SQL)

Этот проект показывает базовую структуру веб‑приложения:

- **Frontend (браузер):** HTML/CSS + один общий файл `public/app.js`
- **Backend (сервер):** `app.js` на Express
- **База данных:** SQLite (файл `database.db`), таблица `users`
- **Авторизация:** сессии (`express-session`) + хеширование паролей (`bcrypt`)

## 1) Что в проекте умеет работать

- **Регистрация** нового пользователя (`/register`)
- **Переход после регистрации** на отдельную страницу (`/registered`)
- **Вход** (`/login`) и создание сессии
- **Личный кабинет** (`/dashboard`)
- **Админ панель** (`/admin`) — доступна только роли `admin`
- **Список пользователей** и **удаление** (только админ)

Важно: **кнопка/ссылка админ панели не показывается обычному пользователю**. Но даже если вручную открыть `/admin`, сервер всё равно не пустит без роли `admin`.

## 2) Требования

- Node.js (лучше 16+)
- npm (ставится вместе с Node.js)

SQLite — это "лёгкая" SQL‑база данных. Она хранится **в одном файле** и не требует отдельного сервера.

## 3) Самое важное про `package.json` и `package-lock.json`

### Что такое `package.json`

`package.json` — это главный файл Node.js‑проекта. В нём описано:

- **Название проекта** (`name`)
- **Версия** (`version`)
- **Скрипты запуска** (`scripts`)
- **Зависимости** (`dependencies`) — какие библиотеки нужны проекту
- **devDependencies** — зависимости только для разработки

Пример из этого проекта:

- `"start": "node app.js"` — обычный запуск
- `"dev": "nodemon app.js"` — запуск в режиме разработки (перезапуск при изменениях)

### Что такое `package-lock.json`

`package-lock.json` создаёт npm автоматически.

Он нужен, чтобы:

- фиксировать **точные версии** всех установленных пакетов
- гарантировать одинаковую установку на другом компьютере

Обычно **его руками не редактируют**.

## 5) База данных SQLite

### 5.1 Где хранится база

База — это файл:

- `database.db`

Он создаётся автоматически при первом запуске.

### 5.2 Как открыть и прочитать базу

Варианты:

- установить расширение в VS Code: **SQLite Viewer** и открыть `database.db`
- использовать программу **DB Browser for SQLite**

Внутри будет таблица `users`.

### 5.3 Таблица `users` создаётся автоматически

При запуске сервера `app.js` создаёт таблицу:

```sql
CREATE TABLE IF NOT EXISTS users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT UNIQUE NOT NULL,
  email TEXT UNIQUE NOT NULL,
  password TEXT NOT NULL,
  role TEXT DEFAULT 'user',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

И создаёт администратора по умолчанию:

- **Логин:** `admin`
- **Пароль:** `admin123`

## 7) Как работает авторизация (простыми словами)

### Регистрация

1. Пользователь заполняет форму `/register`
2. Браузер отправляет `POST /register`
3. Сервер хеширует пароль через `bcrypt`
4. Сервер сохраняет пользователя в SQLite (`database.db`)
5. Браузер переводит пользователя на `/registered`

### Вход

1. Пользователь вводит логин/пароль на `/login`
2. Браузер отправляет `POST /login`
3. Сервер находит пользователя в БД
4. Сервер сравнивает пароль с хешем (`bcrypt.compare`)
5. Если всё ок — сервер записывает в сессию:

- `req.session.userId`
- `req.session.username`
- `req.session.userRole`

### Проверка доступа

В `app.js` есть middleware:

- `isAuthenticated` — проверяет, есть ли `req.session.userId`
- `isAdmin` — проверяет роль `req.session.userRole === 'admin'`

## 10) Почему весь JavaScript вынесен в `public/app.js`

Чтобы:

- не копировать одинаковый код на каждой странице
- править поведение в одном месте
- сохранить функциональность (регистрация/вход/роль/админка)

На каждой странице стоит `data-page="..."`, и `public/app.js` понимает, какая это страница, и подключает нужные обработчики.

## 12) Методика для экзамена (как объяснять устно)

Ниже — заготовка, которую можно проговорить и/или написать.

### 12.1 Зачем вообще нужен backend, если страницы статические

1. **Регистрация и вход** требуют хранения пользователей.
2. Пароли нельзя хранить в браузере в открытом виде.
3. Поэтому нужен сервер (`app.js`), который:

- принимает данные форм
- хеширует пароль
- кладёт пользователя в базу
- создаёт сессию при входе

### 12.2 Почему SQLite — "light SQL"

- это полноценный SQL (таблицы, запросы `SELECT/INSERT/DELETE`)
- не нужен отдельный сервер
- база — один файл `database.db`, его легко показать экзаменатору

### 12.3 Зачем `bcrypt`

- пароль **не сохраняется как текст**
- сохраняется **хеш**
- при входе сравниваем пароль и хеш через `bcrypt.compare`

### 12.4 Зачем `express-session`

- после входа сервер создаёт "коробочку" (сессию)
- в ней хранит `userId`, `username`, `role`
- благодаря этому страницы (`/dashboard`, `/admin`) понимают кто вошёл

### 12.5 Зачем `package.json`

`package.json` нужен, чтобы:

- описать проект и зависимости
- можно было установить всё одной командой: `npm install`
- запускать проект командами из `scripts`:

- `npm start` → `node app.js`
- `npm run dev` → `nodemon app.js`

### 12.6 Зачем `package-lock.json`

- фиксирует точные версии библиотек
- чтобы у преподавателя/на другом ПК проект ставился так же

### 12.7 Почему кнопка админки "не работает" у обычного пользователя

- на фронте мы **прячем/блокируем** кнопку для удобства
- но главное — сервер всё равно проверяет роль и не пустит без `admin`
